# CampusWorld 系统架构设计

## 🏗️ 整体架构
新一代智慧园区OS项目，基于原生AI架构，采用企业级工程化架构设计，借鉴MUD游戏世界的设计原理构筑世界语义，采用全图数据结构构筑知识本体，以LinkSoft为能力底座，实现与物理世界设备，系统，人的交互。

##架构分层设计
作为业务类OS架构分为3层，从下到上分别是：系统适配层（公共服务层），知识与能力层（知识服务与能力服务），Agent服务层；

##领域服务设计
项目包括5个核心领域服务分别是公共服务，知识服务，能力服务，AI使能服务和Agent服务；

###公共服务设计
1、公共服务在设计上提供以下能力：
- IaaS层适配能力，通过Docker化部署或K8s部署实现多环境部署解耦与适配；
- PaaS层适配能力，可支持对接云服务商提供的中间件服务，包括MaaS服务（大模型服务），数据库服务，缓存服务， MQ服务；
- 提供公共能力，包括：认证鉴权，日志，Portal框架，SSH能力等；
- 提供项目工程化能力，包括：安装，部署，升级等

2、知识服务在设计上提供以下能力：
- 基于统一数据库提供各类知识的管理和维护，本项目基于Model驱动理念，借鉴MUD的游戏设计，构建多个虚拟世界（语义的数字孪生），虚拟世界中的各类对象都以数据对象表示，通过分层图数据设计表达对象与对象间的关系
- 知识服务存储各类知识对象也被称为知识节点，知识对象有两大类，一类是概念类，包括：目标（如：实现空间安全），规则（如：本空间超过100人需告警），抽象领域（如： 某虚拟世界），经验（如：某设备大概需要每20天维护），流程（如：员工报销SOP），行动（如：打开某个设备的指令），Skill（如：指导客户服务Agent（某世界的自主NPC）进行自主活动的指导书）等，另外一类是事实类，包括：各类物理空间事实（如：本楼有10层，第一层有30个房间），设备事实（如：某设备长宽高，品牌，保修期），环境感知事实（如：当天的温度，气压；如：某房间的WiFI CSI感知显示房间有人），人员事实（如：本车间当前的操作员是李明）等，部分数据对象本身就是Agent（类似NPC）；
- 知识服务提供知识的查询和推理能力，能够基于要求将符合要求和逻辑的知识（通过本体规则，流程等）构建记忆和语义上下文，为Agent服务层提供知识，实现统一的RAG和OAG；
- 知识服务内置多类知识Worker，知识Worker能够采集，提起，维护，反哺各类知识；

3、能力服务基于LinkSoft构建，LinkSoft当前项目未集成，起提供与各类设备和真实系统集成的能力，包括：设备IoT集成与互动，IT系统（如：ERP）集成与互动，可以说CampusWorld与物理世界交互的能力都需要经过能力服务，能力服务的元数据会被知识服务采集形成语义数字孪生；

4、AI使能服务提供公共的AI能力，包括：对LLM模型的调用，提供内置意图模型，提供安全护栏能力等；

5、Agent服务提供统一的Agent能力框架，包括：Agent构建能力（如：虚拟世界构建，NPC构建），内置的系统Agent（知识服务内置，核心场景如：空间安全NPC，系统维护NPC等），对外提供Agent API（多Agent协同）；

### 多端接入架构

CampusWorld 支持多种客户端接入方式，后端核心逻辑完全共享：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CampusWorld 后端                               │
│                                                                      │
│   ┌──────────────────────────────────────────────────────────────┐   │
│   │                    命令系统 (commands/)                       │   │
│   │  - 命令注册表 (CommandRegistry)                             │   │
│   │  - 授权策略 (CommandPolicyEvaluator)                        │   │
│   │  - 上下文管理 (CommandContext)                              │   │
│   └──────────────────────────────────────────────────────────────┘   │
│                               │                                      │
│         ┌─────────────────────┼─────────────────────┐               │
│         ▼                     ▼                     ▼               │
│   ┌───────────┐       ┌───────────┐          ┌───────────┐         │
│   │ SSH Handler│       │HTTP/WS API│          │ Future API │         │
│   │  (Paramiko)│       │ (FastAPI) │          │  (gRPC等)  │         │
│   └─────┬─────┘       └─────┬─────┘          └─────┬─────┘         │
└─────────┼───────────────────┼──────────────────────┼────────────────┘
          │                   │                      │
          ▼                   ▼                      ▼
   ┌───────────┐       ┌───────────┐        ┌───────────┐
   │ SSH 客户端 │       │CLI 客户端  │        │ Web 客户端 │
   │ (传统终端) │       │(campus)   │        │ (xterm.js) │
   └───────────┘       └───────────┘        └───────────┘
```

| 接入方式 | 协议 | 终端能力 | 适用场景 |
|----------|------|----------|----------|
| SSH | Paramiko/SSH | 行模式（按Enter发送） | 通用终端访问 |
| CLI (`campus`) | WebSocket/HTTP | 即时按键捕获、叠加层 | 增强交互体验 |
| Web | WebSocket | 浏览器内终端 | Web 在线访问 |

### CLI 客户端 (`campus`) 设计

CLI 客户端 `campus` 是一套**与 SSH 完全解耦**的独立客户端，提供增强的终端交互体验：

**与 SSH 的关系**：
- SSH：`app/ssh/` - 基于 Paramiko 的传统 SSH 服务器
- CLI：`client/campus/` - 基于 WebSocket/HTTP 的独立客户端

**核心差异**：
| 特性 | SSH | CLI (`campus`) |
|------|-----|-----------------|
| 终端控制 | 依赖 SSH 协议限制 | 直接控制本地 TTY |
| 即时按键 | ❌ 不支持 | ✅ 支持 |
| 屏幕叠加层 | ❌ 不支持 | ✅ 支持 |
| Tab 补全弹出 | ❌ 不支持 | ✅ 支持 |

**目录结构**：
```
client/campus/           # CLI 客户端（独立包）
├── campus/              # 主包
│   ├── __init__.py
│   ├── __main__.py    # python -m campus
│   ├── config.py       # 配置管理
│   ├── terminal.py     # prompt_toolkit 终端 UI
│   ├── connection.py    # WebSocket 连接
│   ├── protocol.py     # 协议编解码
│   └── completers/     # 补全器
└── pyproject.toml     # 包定义
```

**配置文件** (`~/.config/campus/config.yaml`)：
```yaml
server:
  host: "localhost"
  port: 8000
auth:
  token_file: "~/.config/campus/token"
terminal:
  theme: "default"
completion:
  enabled: true
  show_on_slash: true
```

## SPEC 文档治理规则

- 模块主规范文件固定为：`docs/<module>/SPEC/SPEC.md`、`TODO.md`、`ACCEPTANCE.md`。
- **特性级规范文档必须仅存放在模块的 `features/` 子目录**，命名建议 `Fxx_<FEATURE>.md`。
- 禁止在 `docs/<module>/SPEC/` 根目录新增临时特性 SPEC（避免重复来源和口径分叉）。
- 同一特性只能存在一个规范来源文件；主规范中仅做索引或摘要，不复制完整特性正文。
- 当特性归档或替换时，应更新主规范索引并删除旧的并行特性文档，保持单一事实来源。

## 可插拔虚拟世界包与 HiCampus 数据架构约束

本节描述能力服务侧「世界内容包」在工程上的落点与约束，与上文知识服务中「事实类 / 概念类」知识对象的划分一致：**实体层数据表达物理世界语义化事实，概念层数据表达抽象治理与行为语义**。详细字段、校验层与迁移规则以特性 SPEC 为唯一展开来源。

### 架构角色

- 系统入口（如奇点屋）与世界内容解耦；具体园区/世界由独立包承载，经 `GameLoader` 发现与加载（Evennia 风格：内容可离线维护、运行时装配，避免业务世界写死在系统启动主流程）。
- **F01**：世界包运行时（发现、加载/卸载/重载、控制面持久状态与结构化错误）。
- **F02**：世界独立数据目录与双模型数据包（文件契约、分层校验、world-scoped 迁移）；**不向全局 `seed_minimal` 写入 HiCampus 业务节点**。
- **F03**：将 F02 产出的规范化快照实例化为图节点与关系（语义数字孪生落库）。
- **F01–F03 缺口修复计划**（错误码、loader 挂接种子、关系可见性、Evennia 参照）：[F01_F03_GAP_REMEDIATION_PLAN.md](F01_F03_GAP_REMEDIATION_PLAN.md)

### 包布局（以 HiCampus 为样例）

- 代码与清单：`backend/app/games/hicampus/`
  - `__init__.py`、`game.py`、`manifest.yaml`（必填：`world_id`、`version`、`api_version`、`data_dir`）
  - `package/`：数据包装载与校验（`loader` / `validator` / `migrator` / `contracts`）
- 数据根目录：`backend/app/games/hicampus/data/`（由 `manifest.data_dir` 指向，通常为 `data`）

数据目录分层（**空间 + 实体事实 + 概念抽象**）：

| 层级 | 含义 | 典型文件 |
|------|------|----------|
| Spatial | 世界骨架：世界根、楼栋、楼层、房间、拓扑关系 | `world.yaml`、`buildings.yaml`、`floors.yaml`、`rooms.yaml`、`relationships.yaml` |
| EntityModel | **事实模型**：可定位、可交互的语义实体（NPC/Agent、设备、家具、逻辑区域如安全红区等） | `entities/npcs.yaml`、`entities/items.yaml`、`entities/zones.yaml` |
| ConceptModel | **概念模型**：目标、流程、规则、行为、skills 等抽象约束，经 `bindings` 绑定到实体或空间 | `concepts/goals.yaml`、`processes.yaml`、`rules.yaml`、`behaviors.yaml`、`skills.yaml` |
| 版本与演进 | 包版本元数据与世界内迁移脚本 | `package_meta.yaml`、`migrations/` |

### 工程约束（必须遵守）

1. **与全局种子解耦**：`backend/db/seed_data.py` 仅保留系统最小可运行集；HiCampus 楼栋/房间/实体/概念业务数据**只**存在于世界包 `data/` 下。
2. **图实例化边界**：F02 只产出通过校验的 `PackageSnapshotV2`（或等价规范化结构）；**写库与幂等策略在 F03**，避免控制面与内容面 SQL/ORM 碎片化。
3. **加载前置校验**：世界加载路径应对数据包执行分层校验；失败时返回稳定错误码（如 `WORLD_DATA_UNAVAILABLE`、`WORLD_DATA_INVALID`、`WORLD_DATA_REFERENCE_BROKEN`、`WORLD_DATA_BASELINE_MISMATCH`、`WORLD_DATA_SEMANTIC_CONFLICT` 等），不得拖垮 SSH/会话入口。
4. **关系与导航**：拓扑仍以关系数据表达；双向可通行须在数据层显式维护两条有向关系（与 Evennia Exit 单向性在工程上的对应约定）。
5. **类型与展示**：节点侧沿用 `type_code` + `attributes`（含 `presentation_domains`、`access_locks` 等）扩展，避免为世界业务单独分叉一套持久化 schema。
